This guide covers setting up webhooks in Mailchimp Transactional to receive real-time event notifications — deliveries, opens, clicks, bounces, and spam complaints — posted to a URL you control. Webhooks let your application react to email activity as it happens, without polling the API.

## Available event types

Subscribe to only the events your application needs. Each event type is delivered as a separate object in the webhook payload.

| Event | Trigger |
|---|---|
| `send` | A message is accepted by the API and handed off for delivery. |
| `deferral` | Delivery was attempted but temporarily deferred by the receiving server. |
| `hard_bounce` | The message permanently bounced — the address is invalid. |
| `soft_bounce` | The message temporarily bounced — the mailbox may be full. |
| `open` | The recipient opened the message. |
| `click` | The recipient clicked a tracked link. |
| `spam` | The recipient marked the message as spam. |
| `unsub` | The recipient clicked an unsubscribe link. |
| `reject` | The message was rejected before sending (e.g., on the denylist). |

## Adding a webhook

You can add a webhook from the dashboard or directly via the API.

**From the dashboard:**

<Steps>
  <Step title="Open the Webhooks page">
    Log in to [mandrillapp.com](https://mandrillapp.com/), navigate to **Webhooks**, and click **Add Webhook**.
  </Step>
  <Step title="Enter your endpoint URL">
    Enter the URL where Mailchimp Transactional should POST events. Your endpoint must be publicly reachable over HTTPS.
  </Step>
  <Step title="Select event types">
    Check the event types you want to subscribe to. You can update your subscriptions at any time from the webhook's detail page.
  </Step>
  <Step title="Create and test the webhook">
    Click **Create Webhook**. Verify the setup by sending a test message or triggering an event type via the API and confirming your endpoint receives the payload.
  </Step>
</Steps>

**Via the API:**

```bash
curl -X POST https://mandrillapp.com/api/1.0/webhooks/add \
  -H "Content-Type: application/json" \
  -d '{
    "key": "YOUR_API_KEY",
    "url": "https://yourapp.com/webhooks/mailchimp",
    "events": ["send", "open", "click", "hard_bounce"]
  }'
```

## Webhook payload format

Mailchimp Transactional delivers events as a JSON array in a POST body parameter named `mandrill_events`. Each element in the array represents one event.

```json
[
  {
    "event": "open",
    "ts": 1704067200,
    "_id": "abc123abc123abc123abc123",
    "msg": {
      "email": "jordan@example.com",
      "subject": "Your order has shipped",
      "tags": ["order-shipped"]
    },
    "ip": "203.0.113.42",
    "user_agent": "Mozilla/5.0 (iPhone; CPU iPhone OS 17_0 like Mac OS X)"
  }
]
```

Key fields in every event:

| Field | Description |
|---|---|
| `event` | The event type (e.g., `open`, `click`, `hard_bounce`). |
| `ts` | Unix timestamp of when the event occurred. |
| `_id` | The unique message ID, matching the `_id` returned by the send API. |
| `msg` | An object containing message metadata: recipient email, subject, tags, and more. |

## Responding to webhooks

Your endpoint must return an HTTP `200` status code within **10 seconds** of receiving the request. Mailchimp Transactional interprets any non-200 response — or no response within the timeout — as a failed delivery and will retry.

<Warning>
  If your endpoint is slow or consistently returns non-200 responses, Mailchimp Transactional will retry delivery with exponential backoff and may eventually disable the webhook entirely. Make sure your handler is fast and returns `200` before performing any heavy processing — use a queue to handle work asynchronously.
</Warning>

## Verifying webhook authenticity

Mailchimp Transactional signs each POST request with an HMAC-SHA1 signature so you can confirm the request genuinely originated from Mailchimp. The signature is included in the `X-Mandrill-Signature` HTTP header.

To verify a request:

1. Retrieve your webhook's authentication key from its detail page in the dashboard (or from the `auth_key` field returned by `POST /webhooks/add` or `POST /webhooks/list`).
2. Start with the exact webhook URL you registered — including any query string.
3. Sort the POST parameters alphabetically by key, then append each key immediately followed by its value, with no delimiter between pairs.
4. Compute an HMAC-SHA1 digest of the resulting string using your authentication key as the secret. Use a **binary** (not hexadecimal) digest.
5. Base64-encode the binary digest and compare it to the value in the `X-Mandrill-Signature` header.
6. Reject the request if the values don't match.

```javascript
const crypto = require('crypto');

function generateSignature(webhookKey, webhookUrl, postParams) {
  let signedData = webhookUrl;
  const sortedKeys = Object.keys(postParams).sort();
  sortedKeys.forEach(key => {
    signedData += key + postParams[key];
  });
  return crypto.createHmac('sha1', webhookKey)
    .update(signedData)
    .digest('base64');
}

// Usage: compare result to req.headers['x-mandrill-signature']
```

<Tip>
  Your webhook authentication key is displayed on the webhook's detail page in the dashboard. Treat it as a secret — rotate it if you believe it has been compromised.
</Tip>

## Listing your webhooks

To retrieve all webhooks currently configured on your account:

```bash
curl -X POST https://mandrillapp.com/api/1.0/webhooks/list \
  -H "Content-Type: application/json" \
  -d '{"key": "YOUR_API_KEY"}'
```

The response is an array of webhook objects, each containing the webhook's `id`, `url`, subscribed `events`, and `auth_key`.
