The `/messages/send` endpoint is the core of the API. This guide covers the message object in detail so you can send exactly what your users need.

## The message object

Every send is built around a `message` object. At minimum it needs a sender, a subject, content, and at least one recipient.

```json
{
  "from_email": "orders@example.com",
  "from_name": "Example Store",
  "subject": "Your order has shipped",
  "html": "<p>Hi Jordan, your order #1043 has shipped.</p>",
  "to": [
    { "email": "jordan@example.com", "name": "Jordan Lee", "type": "to" }
  ]
}
```

## Multiple recipients

The `to` array accepts as many recipients as you need, each with a `type` of `to`, `cc`, or `bcc`. Each recipient receives an individual message — recipients never see each other.

```json
"to": [
  { "email": "jordan@example.com", "type": "to" },
  { "email": "records@example.com", "type": "bcc" }
]
```

## Tracking opens and clicks

Set `track_opens` and `track_clicks` to monitor engagement. Tracking data appears per message and can be retrieved with [`/messages/info`](/api-reference/messages/info) or pushed to your systems with [webhooks](/guides/webhooks).

```json
{
  "track_opens": true,
  "track_clicks": true
}
```

## Tagging messages

Tags let you group and report on messages by type. Tag your password resets separately from your receipts, and you can analyze deliverability for each stream independently.

```json
{
  "tags": ["password-reset"]
}
```

## Handling the response

A send returns an array with one result per recipient. Always check `status` per recipient rather than assuming a single outcome:

```json
[
  { "email": "jordan@example.com", "status": "sent", "_id": "abc123" },
  { "email": "invalid@", "status": "invalid", "reject_reason": "invalid-address" }
]
```

| Status | Meaning |
| --- | --- |
| `sent` | Delivered to the receiving server. |
| `queued` | Accepted and waiting to send. |
| `rejected` | Blocked — see `reject_reason`. |
| `invalid` | The address could not be parsed. |
